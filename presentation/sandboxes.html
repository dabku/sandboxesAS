Dlaczego agenty muszą uruchamiać kod

Problem: LLM-y nie potrafią przetwarzać danych

Agenty są wszędzie - ale prawdziwa moc pojawia się wtedy, gdy mogą działać, a nie tylko gadać i pobierać informacje.

  • LLM nie policzy rzetelnie niczego “w głowie” – potrzebuje pandas, numpy, SQL
  • Parsowanie danych wymaga determinizmu – regex, biblioteki, typy danych
  • Prawdziwa praca to interakcja z systemem – pliki, API, bazy danych

Jeśli chcesz agentów o wszechstronnym zastosowaniu, muszą uruchamiać komendy. Ale na czyim komputerze i z jakimi uprawnieniami?

Jak to rozwiązujemy?

LLM-y mają mechanizmy do wywoływania zewnętrznych narzędzi:

  • Function calling / Tool use – LLM zwraca strukturalnie jaką funkcję wywołać i z jakimi argumentami; host ją wykonuje i zwraca wynik
  • MCP (Model Context Protocol) – ustandaryzowany protokół do podłączania narzędzi i źródeł danych do dowolnego modelu; w 2026 jest już szeroko adoptowany w ekosystemie narzędzi dla agentów, i nie, nie został zastąpiony przez skillsy

Wspólny mianownik: agent generuje intencję, a zewnętrzny system ją wykonuje.

Dwa modele zagrożeń

Przez całą prezentację będziemy wracać do dwóch różnych problemów izolacji:

🛡️ Ochrona hosta (host protection)

Niezaufany kod nie może zaszkodzić maszynie, na której działa.

  • Czytanie plików hosta (/etc/passwd, .env)
  • Dostęp do sieci / exfiltracja danych
  • Eskalacja uprawnień, ucieczka z kontenera

🔒 Izolacja tenantów (tenant isolation)

Dane jednego użytkownika/agenta są niewidoczne dla drugiego.

  • Współdzielony filesystem między agentami
  • Wspólne zmienne środowiskowe
  • Brak separacji danych między klientami

Demo 1: Definicja narzędzia (tool)

TOOLS = [
    {
        "type": "function",
        "function": {
            "name": "run_bash",
            "description": (
                "Execute a bash command and return the output. "
                "Use this to run code, analyze files, install packages, etc."
            ),
            "parameters": {
                "type": "object",
                "properties": {
                    "command": {
                        "type": "string",
                        "description": "The bash command to execute",
                    }
                },
                "required": ["command"],
            },
        },
    }
]

To jest surowy schemat JSON wymagany przez OpenAI SDK. W praktyce frameworki agentowe (LangChain @tool, OpenAI Agents SDK @function_tool, Pydantic AI…) pozwalają generować te schematy automatycznie z dekoratorów — łatwiejsze utrzymanie niż ręczny JSON.

Demo 1: Handler narzędzia

def handle_tool(name: str, args: dict) -> str:
    """Dispatch tool calls."""
    if name == "run_bash":
        return run_bash(args["command"])
    return f"Unknown tool: {name}"

Agent zwraca nazwę narzędzia i argumenty → orchestrator wywołuje odpowiednią funkcję → wynik wraca do agenta.

Demo 1: Co robi agent

run_agent(
    system_prompt=(
        "You are a helpful data analyst agent. "
        "You have access to a bash tool to execute commands. "
        "The data files are in the ./sample-data/ directory."
    ),
    user_message=(
        "Analyze ./sample-data/sales_data.csv. "
        "Tell me the total revenue per region. Use pandas."
    ),
    tools=TOOLS,
    tool_handler=handle_tool,
)

Agent pisze kod w pandasie → wywołuje run_bash → dostaje odpowiedź → zwraca wynik.

Demo 5: Docker na uruchomienie - lepiej, ale…

Demo 5: Świeży kontener na każde uruchomienie

Pomysł: Odpalamy świeży kontener Docker na każde wykonanie kodu.

docker_cmd = [
    "docker", "run", "--rm",
    "--network", "none",          # Brak sieci
    "--memory", "256m",           # Limit pamięci
    "--read-only",                # Filesystem tylko do odczytu
    "-v", f"{DATA_DIR}:/data:ro", # Dane zamontowane read-only
    SANDBOX_IMAGE,
    "bash", "-c", command,
]

✅ Co jest dobrze

  • Brak sekretów w kontenerze
  • Czysty filesystem na każde uruchomienie
  • Brak sieci, dane tylko do odczytu

⚠️ Co nie do końca

  • ~0.2-1.2s cold start na każde uruchomienie (zależy od złożoności)
  • Współdzielony kernel hosta (CVE ucieczki)

Demo 5: O czym musimy pamiętać

Podejście “świeży kontener na wywołanie” działa, ale wymaga sporo roboty dookoła:

  • Timeout - co jeśli agent odpali while True?
  • Osierocone kontenery - crash orchestratora = martwe kontenery w tle
  • Zarządzanie obrazami - aktualizacje zależności, wersjonowanie, cache
  • Monitoring i logi - kontenery żyją sekundy, agregacja to osobny problem
  • Brak stanu między krokami - każdy krok to nowy kontener
  • Współdzielony kernel - jedno CVE i izolacja pada

Da się to wszystko zrobić samemu. Pytanie: czy warto, skoro ktoś to już zrobił?

Demo 6: Sandbox Deno

Demo 6: Co to jest Deno?

Deno to środowisko uruchomieniowe dla JavaScript/TypeScript - alternatywa dla Node.js, stworzona przez tego samego autora (Ryan Dahl).

Kluczowa różnica względem Node.js: wbudowany model uprawnień bezpieczeństwa.

  • Domyślnie brak dostępu do systemu plików
  • Domyślnie brak dostępu do sieci
  • Domyślnie brak dostępu do zmiennych środowiskowych
  • Domyślnie brak możliwości uruchamiania podprocesów

Node.js (i Python) domyślnie dają pełny dostęp do systemu. Deno domyślnie odmawia wszystkiego - musisz jawnie zezwolić na każdy zasób.

Demo 6: Model uprawnień w praktyce

deno run \
  --allow-read=./sample-data \   # TYLKO ten katalog
  --deny-net \                   # Brak sieci
  --deny-env \                   # Brak zmiennych środowiskowych
  --deny-run \                   # Brak podprocesów
  --deny-write \                 # Brak zapisu plików
  script.ts

Agent dosłownie nie może zrobić tego, na co mu nie pozwolisz:

  • ❌ Czytać /etc/passwd → PermissionDenied
  • ❌ Zrobić request HTTP → PermissionDenied
  • ❌ Czytać zmienne środowiskowe → PermissionDenied
  • ✅ Czytać ./sample-data/ → Działa!

Demo 7: Zarządzane sandboxy chmurowe

Demo 7: Co to jest Daytona?

Do produkcyjnego multi-tenant można rozważyć zarządzanego sandboxa chmurowego - kategorii narzędzi dających pełną izolację out-of-the-box.

Daytona to platforma do izolowanych środowisk wykonawczych w chmurze. Każdy sandbox to osobne środowisko z własnym filesystemem, procesami i siecią - uruchamiane bardzo szybko (0.1s-0.2s), z gotowego snapshotu lub ciepłej puli.

Dostawcy sandboxów opierają się na różnych modelach izolacji (np. microVM, gVisor, dedykowane kontenery), szczegóły implementacyjne zależą od providera.

Koszt Daytona to $0.0504/h za vCPU i $0.0162/h za GiB RAM, rozliczamy się w sekundach. Jest możliwość self-hosted.

Agent dostaje czysty, izolowany komputer w chmurze - bez ryzyka dla hosta i innych tenantów.

Demo 7: Kilka linijek kodu

from daytona import Daytona, CreateSandboxFromSnapshotParams

daytona = Daytona()
sandbox = daytona.create(
    CreateSandboxFromSnapshotParams(
        language="python",
        auto_stop_interval=3,    # auto-stop po 3 min
        auto_delete_interval=0,  # usunięcie natychmiast
    ),
    timeout=60,
)

result = sandbox.code_interpreter.run_code("print(1 + 1)")
print(result.stdout)  # "2"

sandbox.delete()

Kilka linijek kodu. Sandbox w chmurze. Pełna izolacja. Bez Dockera, bez konfiguracji.

Niezależnie od providera, dostajemy dwie fajne cechy:

✅ Bezpieczeństwo

  • Własny filesystem per sandbox (izolacja tenantów)
  • Zmienne środowiskowe hosta niedostępne (ochrona hosta)
  • Automatyczne czyszczenie

✅ Developer Experience

  • Proste SDK - kilka linijek kodu
  • Szybkie tworzenie (snapshoty / ciepłe pule)
  • Wbudowany interpreter kodu
  • Popularne biblioteki pre-instalowane

Demo 7: Integracja z agentem

# Handler narzędzi agenta - to wszystko
def handle_tool(name: str, args: dict) -> str:
    execution = sandbox.code_interpreter.run_code(args["code"])
    output_parts = []
    if execution.stdout:
        output_parts.append(execution.stdout)
    if execution.error:
        output_parts.append(f"ERROR: {execution.error.value}")
    return "\n".join(output_parts) or "(no output)"

Sandbox provider obsługuje całą infrastrukturę. Ty piszesz tylko logikę agenta.
